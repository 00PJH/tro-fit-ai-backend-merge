# Tro-Fit Vision AI — 핵심 모듈 세부 분석 및 스냅샷(Snapshot) 추출 로직

> **문서 목적:** 파이프라인의 핵심인 `core/` 내부 각 파이썬 코드들의 세부 기능과 구동 원리, 그리고 백엔드 API 응답 속도 최적화의 핵심인 **Snapshot 기반 프레임 추출 로직**의 설계 배경을 상세 분석합니다.
> **수신자:** 조영진 팀원 (시스템 이해 및 후속 파이프라인 제어자)

## 1. Core 모듈 코드별 세부 기능 및 설계 이유

`src/core/` 디렉토리에 위치한 파일들은 전체 시스템의 심장부입니다.

### 1-A. `core/landmark_extractor.py` (MediaPipe 래퍼 및 프레임 추출)
**기능:**
- 비디오 프레임 BGR 이미지를 MediaPipe RGB 포맷으로 변환하고, 모델을 추론합니다.
- `IMAGE 모드`와 `VIDEO 모드`용 객체 생성기를 각각 제공합니다. (조영진 님의 `detect_for_video` 로직 흡수)

**왜 이렇게 구현했나?**
MediaPipe 자원 할당과 해제를 잊어버려 생기는 메모리 릭(Memory Leak)을 막기 위해 Python의 `with` 구문(Context Manager)을 사용하도록 감쌌습니다. 조영진님의 코드에서도 잘 구현되어 있던 자원 해제 기법을 강제하여 서버 안정성을 확보했습니다.

### 1-B. `core/angle_engine.py` (3D 각도 연산 코어)
**기능:**
- 파이프라인에서 추출된 좌표를 받아 `calculate_angle_3d` 함수에서 3D 벡터 내적으로 관절 각도를 도출합니다.
- 각 관절 계산 함수들(`compute_shoulder_angles` 등)이 구현되어 있습니다.

**핵심 로직: `_compute()` 함수의 Visibility 게이트**
```python
# angle_engine.py 내부 로직 발췌
if min(lm_a.visibility, lm_v.visibility, lm_c.visibility) < threshold:
    return res.reliable = False
```
**왜 이렇게 구현했나?** 
관절 각도는 A, V, C 세 점을 이용해 구합니다. 만약 팔꿈치가 화면 밖으로 나가 가시성(Visibility) 점수가 떨어졌는데 무리하게 좌표를 추정해 각도를 구하면, 말도 안 되는 ROM 수치가 나와 환자에게 혼란을 줍니다. 세 점 중 **단 하나라도** 임계값(0.65) 밑으로 떨어지면 그 각도 계산을 과감히 포기(`reliable: false`)하도록 가장 엄격한 보수적 기준을 채택한 것입니다.

### 1-C. `core/visualizer.py`
**기능:**
- 단순히 점을 찍고 선을 그리는 것을 넘어, 각 관절 위치에 "Knee: 110.7deg" 형태의 노란색 텍스트 오버레이를 씌웁니다.
- 글자가 겹치지 않도록 자동 회피하는 Bounding Box 클리핑 알고리즘이 내장되어 있습니다.

## 2. Snapshot 프레임 추출 로직 심층 분석 (가장 중요한 성능 최적화 포인트)

기본적으로 `main.py`를 실행할 때 쓰이는 `pipeline/snapshot_rom_pipeline.py`의 핵심 로직입니다. 비디오 전체를 분석하지 않고 **특정 3장만 뽑아내는 로직**입니다.

### 2-A. 왜 전체 프레임(Full Frame)을 분석하지 않는가?
- **연산 병목 해소:** 4초짜리 30FPS 비디오는 총 120프레임입니다. 서버에서 120장을 전부 추론하고 연산하면 Latency(지연시간)가 수십 초 이상 걸릴 수 있습니다. 사용자가 운동 영상을 올리고 결과를 확인하기까지 로딩창을 무한정 보게 만들 수 없습니다.
- **비즈니스 목적:** 우리 서비스가 집중하는 것은 "가장 많이 굽혔을 때 각도(Max) - 처음 가만히 있을 때 각도(Neutral)"인 ROM의 차이값입니다. 중간 100프레임의 궤적은 DB에 넣을 필요도 없는 잉여 데이터입니다.

### 2-B. 스냅샷(Snapshot) 추출 로직 및 후보 선택 알고리즘

```python
# snapshot_rom_pipeline.py 로직 흐름
1. 중립 캡처: 영상 시작 후 0.5초 부분을 강제로 읽음 (Neutral Frame)
2. 후보 캡처: 영상 끝-1.0초 지점과 끝-0.5초 지점을 강제로 읽음 (Max Candidates)
3. 베스트 선택: 후보 2장 중 Visibility와 극단성 점수를 비교해 가장 우수한 1장 채택
```

**동작 방식 상세:**
OpenCV의 `cap.set(cv2.CAP_PROP_POS_FRAMES, n)` 기능을 이용합니다. 이는 비디오를 1번부터 n번 프레임까지 순차적으로 압축을 풀며 읽는 것이 아니라, O(1) 시간복잡도로 타겟 프레임으로 점프(Random Access)하여 딱 한 장만 읽고 바로 닫아버리는 극도의 최적화 기법입니다.

**어떻게 2개의 최대 자세 후보 중 '최적'을 고르는가? (`select_best_max_frame`)**
1. **가시성(Visibility) 점수 비교:** 후보 A와 B의 관절 가시성 합산을 비교합니다. 둘 중 감지가 실패했거나 신뢰도가 너무 떨어지는 것이 있다면 즉각 탈락시킵니다.
2. **자세 극단성(Extremeness) 점수 비교:** 둘 다 가시성이 훌륭하다면, 사용자가 더 많이 동작을 수행한(더 극단적으로 각도가 꺾인) 프레임을 선택합니다. 우리는 `각도의 역수의 합` 알고리즘을 사용하여, 어깨나 무릎이 더 많이 접혀서 각도가 낮아질수록 점수를 높게 주도록 수학적으로 설계했습니다.
